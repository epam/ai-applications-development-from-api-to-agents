→ REQUEST

GET /.well-known/oauth-authorization-server Host: localhost:8007

← RESPONSE

200 OK Content-Type: application/json { "issuer": "https://auth.example.com", "authorization_endpoint": "https://auth.example.com/authorize", "token_endpoint": "https://auth.example.com/oauth/token", "jwks_uri": "https://auth.example.com/.well-known/jwks.json", "scopes_supported": ["mcp:tools", "mcp:read"], "response_types_supported": ["code"], "code_challenge_methods_supported": ["S256"] }
Why this matters: The MCP client does NOT hardcode the auth server URL. It discovers it dynamically from the MCP server. This means you can swap auth providers without changing client code.

Auth Server validates:   code is valid & not expired  |  code_verifier hashes to stored code_challenge  |  client_id matches

→ TOKEN REQUEST

POST /oauth/token Content-Type: application/x-www-form-urlencoded grant_type=authorization_code &code=SplxlOBeZQQYbYS6WxSbIA &redirect_uri=http://localhost:9999/callback &client_id=my-mcp-client &code_verifier=dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk

← TOKEN RESPONSE

200 OK Content-Type: application/json { "access_token": "eyJhbGciOiJSUzI1NiJ9…", "token_type": "Bearer", "expires_in": 3600, "refresh_token": "v1.MbU2HT4…", "scope": "mcp:tools" }
PKCE security: Even if the authorization code is intercepted, the attacker can't exchange it — they don't have the code_verifier. This is why PKCE replaces client secrets for public clients like CLI tools.
